# Knowledge

- `cargo test -p meshc --test e2e e2e_migration -- --nocapture` only runs tests whose names contain the `e2e_migration` substring. If you add migration e2e coverage in `compiler/meshc/tests/e2e.rs`, keep that prefix in the test name or the slice/task gate will silently skip the new proof.
- `reference-backend` startup is stable on the non-empty `DATABASE_URL` path when env validation stays local to `main.mpl` and uses `Env.get` plus `Env.get_int`; the mechanical regression surface is `cargo test -p meshc --test e2e_reference_backend e2e_reference_backend_runtime_starts -- --ignored --nocapture`, and the missing-env check remains `env -u DATABASE_URL PORT=18080 JOB_POLL_MS=500 ./reference-backend/reference-backend 2>&1 | rg "DATABASE_URL"`.
- `reference-backend` worker recovery now depends on two timing rules together: a watchdog actor respawns the worker only after `tick_age_ms` goes stale for `processing`/`crashing` states, and the restarted worker pauses for `recovery_reclaim_grace_ms(poll_ms)` after reclaim so the Rust harness can actually observe the degraded/recovering window before the requeued job is processed again. If the crash-recovery proofs regress, inspect those two timings before touching docs, deploy smoke, or the Postgres reclaim SQL.
- `meshc migrate` currently parses as `meshc migrate [DIR] [COMMAND]`, so project-scoped calls must be ordered like `cargo run -p meshc -- migrate reference-backend status` and `cargo run -p meshc -- migrate reference-backend up`, not `migrate status reference-backend`.
- Compiled Mesh code expects unwrapped runtime `Result`/`Option` payloads for opaque handles and parsed primitives to be boxed payload pointers, not raw integer-as-pointer values. The S01 fixes that unblocked the golden path were: box `PoolHandle` values returned from `mesh_pool_open`/`mesh_pool_checkout`, box row counts returned from `mesh_pg_execute`, and box numeric payloads returned from `String.to_int` / `String.to_float`.
- If `meshc migrate reference-backend up` fails at `mesh_pool_checkout` or the worker crashes in `Storage.Jobs.parse_attempts`, inspect runtime payload boxing and compiled-Mesh unwrap/codegen assumptions before blaming pool opening or SQL itself.
- In `reference-backend`, the generated `Job.__table__()` schema helper collides badly enough with Mesh's own `Job` surface that using the concrete table name (`"jobs"`) is the safer local option.
- The compiler-facing runtime-start proof is sensitive to stale listeners on `:18080`; clear any leftover `reference-backend` process before rerunning `e2e_reference_backend_runtime_starts` or the test can probe the wrong process and report missing startup logs.
- The canonical compiler e2e invocations for `reference-backend` should use Cargo's working argument order: `cargo test -p meshc --test e2e_reference_backend <test_name> -- ...`. Putting the test-name filter before `--test` can make Cargo misparse the request and claim the target does not exist even though `compiler/meshc/tests/e2e_reference_backend.rs` is present.
- `cargo test -p meshc --test e2e_reference_backend e2e_reference_backend_multi_instance_claims_once -- --ignored --nocapture` is now the authoritative shared-DB exact-once gate and should run **1 test**. If it ever exits 0 while running 0 tests again, treat that as a harness regression rather than as evidence that multi-instance correctness passed.
- In this worktree, the real Postgres-backed `meshc` verification commands depend on the repo-root `.env`, but non-interactive shell runs do **not** inherit it automatically. Source `.env` (or load it into the subprocess env) before running `cargo test -p meshc --test e2e_reference_backend ...`, otherwise `${DATABASE_URL:?set DATABASE_URL}` guards fail before the actual runtime proof runs.
- For `cargo test -p meshc --test e2e_reference_backend ...` in this environment, prefer the blocking `bash` tool over `async_bash` when you need authoritative evidence. The async wrapper intermittently reported `no test target named e2e_reference_backend` even though `cargo metadata` and direct blocking runs saw the target correctly.
- For the two-instance `reference-backend` proof, treat `/health.failed_jobs` and `/health.last_error` as the stable shared-DB contention signal, but prove exact-once and cross-instance participation through direct `jobs` table reads, `/jobs/:id`, and per-instance `Job worker processed id=` logs. Summing `/health.processed_jobs` or trusting `/health.last_job_id` under heavier multi-instance polling can introduce harness flake without changing the underlying DB truth.
- `reference-backend/storage/jobs.mpl` should keep `claim_next_pending_job()` as one atomic `Repo.query_raw(...)` claim-and-return statement using `FOR UPDATE SKIP LOCKED`; reverting to a read-then-update flow reintroduces benign shared-DB claim races as false worker failures (`failed_jobs` growth / `update_where: no rows matched`).
- For `meshc test <path>`, treat the requested path as the **discovery root** (project root, `tests/` directory, or specific `*.test.mpl` file), but keep the **import-copy root** anchored at the nearest ancestor containing `main.mpl`. That split is what makes directory-target execution work without regressing project-local imports.
- In `compiler/mesh-fmt/src/walker.rs`, `walk_type_alias_def(...)` must add an explicit separator after a `VISIBILITY` node. If it falls through to the generic inline walker, `pub type Foo = Bar` collapses to `pubtype Foo = Bar`, and `meshc fmt` can turn valid exported type aliases into parse-invalid source.
- In `compiler/mesh-lsp/src/analysis.rs`, backend-shaped editor sessions need project-aware analysis from the nearest ancestor containing `main.mpl` plus open-document overlays. Falling back to single-file analysis can surface bogus import failures or stale diagnostics even when the live buffer is correct.
- Treat `meshc test --coverage <path>` returning exit code 1 with the explicit unsupported message as the authoritative S03 contract. A green placeholder or success-without-report should be treated as a tooling-truth regression, not as partial coverage support.
- For deploy-path diagnostics, `reference-backend/scripts/apply-deploy-migrations.sh` should check the SQL artifact path before checking `DATABASE_URL`; otherwise the failure surface collapses to a generic env error and hides the more actionable missing-artifact cause that S04 verification expects.
- In Mesh source, `case` arms only allow multi-statement bodies via explicit `-> do ... end`; a bare newline after `->` is still parse-invalid. If future language work wants to remove the `let _ = ...` dogfood noise around branch-local side effects, the parser path to inspect is `compiler/mesh-parser/src/parser/expressions.rs::parse_match_arm` (and the analogous arrow-body closure parsing).
- Do not use `mesh_panic`-driven failures (for example non-exhaustive matches) or stdlib panics like `List.head([])` to simulate supervisor child crashes in Mesh actors right now: they currently abort the whole process with `panic in a function that cannot unwind` instead of yielding a supervisor-restartable actor failure. Resume S05 by making the worker exit cooperatively and letting the permanent supervisor restart it, or by adding a real actor-exit-with-error surface to the language/runtime.

- The S08 stale-claim sweeps are literal `rg` checks. Do not write sentences like "no placeholder" or "not a current blocker" into reconciled artifacts, because the negative grep still matches those substrings even when the sentence is negating them.

The build can fail with false missing-module errors (`minisearch`, `@rollup/rollup-darwin-arm64`) while `node_modules` is still being replaced. - `e2e_reference_backend_worker_crash_recovers_job` can pass in isolation while still failing in the full serial closeout gate. For M028 closure, the authoritative signal is the serial acceptance sequence (build/fmt/test/migrate/smoke -> migration truth -> multi-instance truth -> worker-crash recovery), not the isolated recovery test alone.
- In `reference-backend/jobs/worker.mpl`, recovery timing has two opposing constraints: the degraded/recovering window must stay visible long enough for the harness to observe the requeued `pending` row, and the restarted worker must keep ticking long enough afterward for `/health` to report a healthy processed state instead of going stale. If the crash-recovery gate flakes after a seemingly good fix, inspect `crash_restart_delay_ms(...)`, `pause_after_recovery(...)`, and post-recovery tick cadence together rather than treating them as independent knobs.
- In the parser's CST, LITERAL nodes inside multiline arg lists can have NEWLINE trivia tokens as leading children. Any AST accessor that iterates `children_with_tokens()` looking for a meaningful token must filter trivia (`.find(|t| !t.kind().is_trivia())`) instead of using `.next()`. The `Literal::token()` fix in M031/S01/T03 protects `infer_literal`, `lower_literal`, and all future callers. If similar bugs appear for other AST node types with `.next()` patterns, check for the same trivia-leaking issue.
- When a typechecker inference function (like `infer_if`) is called recursively and bypasses `infer_expr`, it must store its resolved type in the `types` map before returning. Without this, codegen's `resolve_range` finds nothing and falls back to `MirType::Unit`. The sentinel for this class of bug is String-return branches — they crash on misaligned pointer dereference when the type is missing, while Int branches silently return garbage.
- The `suppress_trailing_closure` parser flag uses a save/restore pattern around `expr(p)` calls in control-flow condition sites. If a new control-flow form is added to the parser that takes a condition expression before `do`, it must save/set/restore this flag or `fn_call() do` will be misparsed in that position.
- 10 pre-existing `try_*`/`from_try_*` e2e tests fail with runtime crashes (exit code None) on the current `main` branch. These are unrelated to M031/S01 fixes and should not be confused with regressions.
- In the parser's CST, `L_PAREN`/`R_PAREN` tokens for parenthesized imports live inside `IMPORT_LIST`, not as direct children of `FROM_IMPORT_DECL`. Formatter or analysis code that needs to detect paren-wrapped imports must inspect children of `IMPORT_LIST`, not `FROM_IMPORT_DECL`. The `walk_import_list` function in `compiler/mesh-fmt/src/walker.rs` is the reference implementation.
- The parser's existing `paren_depth` mechanism (which makes newlines insignificant inside parens) can be reused for new delimiter-wrapped syntactic forms. `parse_from_import_decl` bumps paren depth on `L_PAREN` so multiline import lists parse without a new parser mode.
- `meshc fmt` does not preserve parenthesized multiline imports: the formatter collapses them back to single-line and introduces spurious spaces in module dot-paths (e.g., `Storage. Schema` instead of `Storage.Schema`). Until the formatter walker is fixed to handle `FROM_IMPORT_DECL` with `IMPORT_LIST` inside parens, `fmt --check` will always fail on mesher (35 files). The parser supports multiline imports correctly (confirmed by `e2e_multiline_import_paren_multiline` test), so the issue is purely in `compiler/mesh-fmt/src/walker.rs`.
- `do...end` blocks cannot be used as values in `let` bindings in Mesh (e.g., `let x = do ... end` is invalid). For block-as-expression semantics, use `if/else` or `case` arms instead. This affects test design but not real-world usage.
