# Knowledge

- `reference-backend` startup is stable on the non-empty `DATABASE_URL` path when env validation stays local to `main.mpl` and uses `Env.get` plus `Env.get_int`; the mechanical regression surface is `cargo test -p meshc --test e2e_reference_backend e2e_reference_backend_runtime_starts -- --ignored --nocapture`, and the missing-env check remains `env -u DATABASE_URL PORT=18080 JOB_POLL_MS=500 ./reference-backend/reference-backend 2>&1 | rg "DATABASE_URL"`.
- `meshc migrate` currently parses as `meshc migrate [DIR] [COMMAND]`, so project-scoped calls must be ordered like `cargo run -p meshc -- migrate reference-backend status` and `cargo run -p meshc -- migrate reference-backend up`, not `migrate status reference-backend`.
- Compiled Mesh code expects unwrapped runtime `Result`/`Option` payloads for opaque handles and parsed primitives to be boxed payload pointers, not raw integer-as-pointer values. The S01 fixes that unblocked the golden path were: box `PoolHandle` values returned from `mesh_pool_open`/`mesh_pool_checkout`, box row counts returned from `mesh_pg_execute`, and box numeric payloads returned from `String.to_int` / `String.to_float`.
- If `meshc migrate reference-backend up` fails at `mesh_pool_checkout` or the worker crashes in `Storage.Jobs.parse_attempts`, inspect runtime payload boxing and compiled-Mesh unwrap/codegen assumptions before blaming pool opening or SQL itself.
- In `reference-backend`, the generated `Job.__table__()` schema helper collides badly enough with Mesh's own `Job` surface that using the concrete table name (`"jobs"`) is the safer local option.
- The compiler-facing runtime-start proof is sensitive to stale listeners on `:18080`; clear any leftover `reference-backend` process before rerunning `e2e_reference_backend_runtime_starts` or the test can probe the wrong process and report missing startup logs.
- The canonical compiler e2e invocations for `reference-backend` should use Cargo's working argument order: `cargo test -p meshc --test e2e_reference_backend <test_name> -- ...`. Putting the test-name filter before `--test` can make Cargo misparse the request and claim the target does not exist even though `compiler/meshc/tests/e2e_reference_backend.rs` is present.
- `cargo test -p meshc --test e2e_reference_backend e2e_reference_backend_multi_instance_claims_once -- --ignored --nocapture` is now the authoritative shared-DB exact-once gate and should run **1 test**. If it ever exits 0 while running 0 tests again, treat that as a harness regression rather than as evidence that multi-instance correctness passed.
- In this worktree, the real Postgres-backed `meshc` verification commands depend on the repo-root `.env`, but non-interactive shell runs do **not** inherit it automatically. Source `.env` (or load it into the subprocess env) before running `cargo test -p meshc --test e2e_reference_backend ...`, otherwise `${DATABASE_URL:?set DATABASE_URL}` guards fail before the actual runtime proof runs.
- For the two-instance `reference-backend` proof, treat `/health.failed_jobs` and `/health.last_error` as the stable shared-DB contention signal, but prove exact-once and cross-instance participation through direct `jobs` table reads, `/jobs/:id`, and per-instance `Job worker processed id=` logs. Summing `/health.processed_jobs` or trusting `/health.last_job_id` under heavier multi-instance polling can introduce harness flake without changing the underlying DB truth.
- `reference-backend/storage/jobs.mpl` should keep `claim_next_pending_job()` as one atomic `Repo.query_raw(...)` claim-and-return statement using `FOR UPDATE SKIP LOCKED`; reverting to a read-then-update flow reintroduces benign shared-DB claim races as false worker failures (`failed_jobs` growth / `update_where: no rows matched`).
- For `meshc test <path>`, treat the requested path as the **discovery root** (project root, `tests/` directory, or specific `*.test.mpl` file), but keep the **import-copy root** anchored at the nearest ancestor containing `main.mpl`. That split is what makes directory-target execution work without regressing project-local imports.
- In `compiler/mesh-fmt/src/walker.rs`, `walk_type_alias_def(...)` must add an explicit separator after a `VISIBILITY` node. If it falls through to the generic inline walker, `pub type Foo = Bar` collapses to `pubtype Foo = Bar`, and `meshc fmt` can turn valid exported type aliases into parse-invalid source.
- In `compiler/mesh-lsp/src/analysis.rs`, backend-shaped editor sessions need project-aware analysis from the nearest ancestor containing `main.mpl` plus open-document overlays. Falling back to single-file analysis can surface bogus import failures or stale diagnostics even when the live buffer is correct.
- Treat `meshc test --coverage <path>` returning exit code 1 with the explicit unsupported message as the authoritative S03 contract. A green placeholder or success-without-report should be treated as a tooling-truth regression, not as partial coverage support.
